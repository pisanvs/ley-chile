# Incremental Pipeline Design — GitHub Actions + Ordered Historial

**Date:** 2026-05-25
**Status:** Approved

## Problem

The current 4-phase pipeline runs end-to-end in a single process. Fetching ~100k normas and
their versioned XML content takes many hours — exceeding GitHub Actions' 6-hour job limit.
If a run dies partway through, all in-memory progress is lost and the run must restart from
scratch.

## Goals

1. **Time limits** — split work across multiple GH Actions runs over many days.
2. **Restart resilience** — a killed run loses at most a few minutes of work; the next run
   picks up exactly where it left off.
3. **Strict ordering** — `historial` is always in a valid, fully-ordered state after each
   push. Commits are never emitted for a date range unless ALL normas in that range are
   fully cached.
4. **Shareability** — a live progress bar in `README.md` reflects pipeline state after
   every run.

## Non-Goals

- Parallelising fetches across multiple concurrent GH Actions runners (out of scope for now).
- Changing the `historial` commit format or the `CommitContext` model.
- Using external storage (S3, separate DB, etc.).

---

## Branch Architecture

Three orphan branches in the same repository, following the existing `historial` worktree
pattern:

```
main (code branch)
├── scripts/                          unchanged
├── graph.json                        NEW: committed here after phases 1+2
├── catalog.json                      NEW: committed here after phase 1
└── .github/workflows/
    ├── update-graph.yml              NEW: rebuilds graph.json
    └── fetch-and-build.yml           NEW: incremental fetch + historial advance

pipeline-cache  (orphan)
└── cache/
    ├── normas/{id}.json
    ├── versions/{id}/{fecha}.json
    └── diffs/{id}.json

historial  (orphan, unchanged)
└── leyes/, modificaciones/, dl/, …
```

`graph.json` is promoted from a local runtime artifact to a committed, version-controlled
input. It is regenerated by `update-graph.yml` when new laws are published (or manually).

`pipeline-cache` is checked out as a git worktree at `./cache` during each run, exactly
like `historial` is checked out at `./historial`.

`historial`'s last commit date serves as the **progress watermark** — no separate progress
file is needed.

---

## Run Lifecycle (`fetch-and-build.yml`)

Each scheduled run (e.g. every 6 hours) executes two sequential steps.

### Step 1 — Fill the cache

```
graph.json  →  identify uncached normas (sorted by fechaPublicacion)
            →  for each: estimate cost = len(versions in diffs/{id}.json)
            →  accumulate normas until VERSION_BUDGET is exhausted
            →  fetch their versions from LeyChile (fetch_versions.py)
            →  commit to pipeline-cache every COMMIT_GRANULARITY normas (e.g. 50)
            →  push pipeline-cache
```

The **budget is in total version-fetches**, not wall-clock time (e.g. `VERSION_BUDGET=20000`).
This is stable across machines and independent of network variance.

`diffs/{id}.json` is fetched first (cheaply) to know a norma's version count before deciding
whether it fits in the budget.

Commits happen every 50 normas so a killed run loses at most ~50 normas of work.

### Step 2 — Advance the watermark

```
pipeline-cache  +  historial (last commit date = W)
→  compute D = max date where cache is complete for all normas ≤ D
→  if D > W:
      build_history.py --append --from W --to D
      push historial
→  update README.md progress block
→  commit + push README.md to main
```

If `D == W` (cache didn't advance far enough to unlock a new complete window), `historial`
is not touched. The README is still updated to reflect cache progress.

---

## Ordering Guarantee

The watermark `D` is computed as:

```python
# For every norma in graph sorted by fechaPublicacion:
#   check cache/diffs/{id}.json exists
#   check cache/versions/{id}/{fecha}.json exists for every version listed in diffs
# D = max date where this holds contiguously from the beginning
```

"Contiguously from the beginning" means: if norma 1234 (dated 1965-03-10) is missing,
`D` cannot advance past 1965-03-09 regardless of how many later normas are cached.

`build_history.py --from W --to D` only emits commits for causing normas with
`W < fechaPublicacion ≤ D`. Since all normas in that window are guaranteed cached, no
commit is ever emitted with missing content.

`git fast-import` is atomic: `historial` either advances cleanly or stays unchanged.

---

## Dynamic Batch Sizing

Each run processes the next N uncached normas in chronological order, where N is determined
by accumulating version-fetch cost until `VERSION_BUDGET` is hit. This means:

- Runs with many small normas process more of them.
- Runs with a few large normas (many versions) process fewer.
- The run never over-commits to more work than it can complete.

`VERSION_BUDGET` is a configurable workflow input (default: `20000`). It can be raised for
manually-triggered catch-up runs.

---

## Resilience Properties

| Failure mode | Effect | Recovery |
|---|---|---|
| Run killed mid-fetch | Lose at most 50 normas of in-flight work | Next scheduled run continues from last committed cache state |
| `build_history.py` crashes | `historial` unchanged (fast-import is atomic) | Fix bug, re-run; watermark re-computed correctly |
| `push pipeline-cache` fails | Local work lost | Next run re-fetches the uncommitted batch |
| Graph is stale | Watermark may stall at a pre-stale date | Trigger `update-graph.yml` manually |

## README Progress Bar

At the end of each `fetch-and-build.yml` run, a Markdown block in `README.md` between
`<!-- PIPELINE_STATUS_START -->` and `<!-- PIPELINE_STATUS_END -->` comment markers is
regenerated and committed to `main`:

```markdown
## Pipeline Status
| | |
|---|---|
| **Historial** | `████████████░░░░░░░░░` 58% · watermark 1997-03-15 · 49,231 normas |
| **Cache**     | `████████████████░░░░░` 74% · 62,800 / 85,000 normas fetched |
| **Last run**  | 2026-05-24 18:42 UTC · next: scheduled |
```

No badge server required. The block is replaced in-place using the sentinel comments so
only the status section changes each commit.

## Script Changes Required

### Modified scripts

| Script | Change |
|---|---|
| `fetch_versions.py` | Add `--version-budget N` flag (budget-based stopping); already skips cached files |
| `build_history.py` | Add `--from` / `--to` date flags to bound which commits are emitted |

### New scripts / modules

| File | Purpose |
|---|---|
| `scripts/compute_watermark.py` | Given `graph.json` + `pipeline-cache/`, returns `(W, D)` |
| `scripts/update_readme_status.py` | Regenerates the progress block in `README.md` |
| `.github/workflows/update-graph.yml` | Runs phases 1+2, commits `graph.json` to main |
| `.github/workflows/fetch-and-build.yml` | Incremental fetch + watermark advance + README update |

### Unchanged

`build_catalog.py`, `fetch_normas.py`, `utils.py`, `enrichers/` — no changes needed.
`run_pipeline.py` remains for local full-pipeline runs.

## Workflow Inputs (`fetch-and-build.yml`)

| Input | Default | Description |
|---|---|---|
| `version_budget` | `20000` | Max version fetches per run |
| `commit_granularity` | `50` | Commit cache every N normas |
| `notify_url` | `""` | ntfy.sh webhook for completion notification |
| `dry_run` | `false` | Fetch + compute watermark but don't push |
